# 1. 코딩 전에 먼저 생각하기

가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러낼 것.

구현 전에:

- 가정이 있다면 명시적으로 밝힌다.
- 구현에 영향을 주는 핵심 정보가 없으면 질문한다.
- 해석이 여러 가지라면 모두 제시한다.
- 혼자 임의로 선택하지 않는다.
- 구현 방식만 여러 가지인 경우 가능한 옵션을 제시하고 가장 단순한 방식을 기본안으로 제안한다.
- 더 단순한 방법이 있다면 제안한다.

### Peakda 적용 예시

- TourAPI 응답 구조가 불명확하면 → 실제 응답 확인 후 진행
- 카카오맵 마커 클릭 처리 방식이 여러 가지라면 → 옵션 제시 후 결정 요청
- 필터 상태를 URL 쿼리로 할지 Zustand로 할지 모호하면 → 질문 후 진행

# 2. 단순함 우선

요청한 것만 구현한다.

- 요청하지 않은 기능은 추가하지 않는다.
- 단일 사용 코드에 불필요한 추상화를 만들지 않는다.
- 요청하지 않은 유연성·확장성을 넣지 않는다.
- 발생 가능성이 낮은 시나리오를 위한 코드를 미리 작성하지 않는다.
- 더 적은 코드로 동일한 결과를 만들 수 있다면 단순한 방식을 선택한다.

### 자문

"시니어 개발자가 보면 과하다고 할까?"

Yes라면 단순화한다.

---

# 3. 최소 변경 원칙

꼭 필요한 것만 수정한다.

- 인접한 코드·주석·포맷을 개선하지 않는다.
- 안 깨진 코드는 리팩토링하지 않는다.
- 기존 코드 스타일을 따른다.
- 관련 없는 데드코드는 건드리지 않는다.

내 변경으로 인해 사용되지 않게 된 경우만 정리한다.

- 사용되지 않는 import 제거
- 사용되지 않는 변수 제거
- 사용되지 않는 함수 제거

기준:

변경된 모든 줄은 요청 사항과 직접 연결되어야 한다.

---

# 4. 목표 기반 실행

작업을 검증 가능한 목표로 변환한다.

예시:

- 필터 추가
  → 꽃 종류·상태 선택 시 지도 마커가 필터링되어야 함

- 버그 수정
  → 마커 클릭 시 BottomSheet가 정상적으로 열려야 함

- 리팩토링
  → 수정 전후 동일하게 동작해야 함

여러 단계 작업은 간략한 계획을 먼저 제시한다.

| 항목            | 규칙                                   |
| --------------- | -------------------------------------- |

| 외부 API        | Route Handler 프록시 경유              |
| 카카오맵        | `dynamic import + ssr: false`          |
| 로그            | `console.error`, `console.warn`만 허용 |

# 코드 생성 규칙

- 사용하지 않는 useEffect 생성 금지
- 사용하지 않는 useMemo 생성 금지
- 사용하지 않는 useCallback 생성 금지
- 단순 상태 공유를 위해 Context를 사용하지 않는다
- 타입 단언(as)을 최소화한다
- 기존 프로젝트 패턴이 있다면 우선 따른다

---

# React

- 파생 가능한 값은 state로 저장하지 않는다
- state는 최소화한다
- useEffect는 외부 시스템 동기화 용도로만 사용한다
- useEffect 내부에서 데이터 가공을 하지 않는다
- Props drilling보다 Context를 우선하지 않는다
- Context 사용 전 Zustand 사용 가능 여부를 검토한다

---

# Next.js

- Server Component를 기본으로 사용한다
- 필요한 경우에만 `use client`를 추가한다
- 가능하면 Server Component 또는 Route Handler에서 데이터를 가져온다
- Client Component 내부 fetch는 필요한 경우에만 사용한다
